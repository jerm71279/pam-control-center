Lifecycle Phases
P1
Discovery & Dependency Mapping
Scan CyberArk on-prem to inventory accounts, safes, platforms, and NHIs
Agents 11, 01, 09, 12 +
Agent 11 — Source Adapter
  • Normalize accounts from any source vendor (CyberArk, BeyondTrust, HashiCorp, AWS, Azure, GCP) into canonical format
  • Provides unified NormalizedAccount schema regardless of source PAM vendor
Agent 01 — Discovery
  • Connect to CyberArk PVWA via GET /Accounts (paginated, count-based)
  • Enumerate all safes via GET /Safes — filter system safes (System, VaultInternal, Notification Engine, etc.)
  • Enumerate platforms via GET /Platforms/Targets
  • NHI Detection (multi-signal): platform-based (UnixSSHKeys, WinServiceAccount, AzureServicePrincipal), name patterns (^svc[_-], ^app[_-], ^api[_-]), safe name patterns (appcred, automation, cicd)
  • Scan for CCP/AAM integrations via GET /Applications + GET /Applications/{id}/Authentications
  • Output discovery manifest with account counts, safe inventory, NHI list, integration map, and platform inventory
Agent 09 — Dependency Mapper
  • Build dependency graph: application → account → safe → platform mappings
  • Scan IIS logs and config files for CCP/AAM connection strings
  • Identify NHI dependency chains that must migrate together
  • Produce migration ordering constraints for Agent 04 Migration Pipeline wave classification
Agent 12 — NHI Handler
  • Weighted multi-signal NHI classification: platform (0.35), name pattern (0.20), container context (0.10), dependency graph (0.25), audit frequency (0.10)
  • Risk scoring: confidence threshold (default 0.6) determines NHI vs human classification
  • Generate rotation policies based on NHI type (service account, API key, certificate)
P1
Gap Analysis & Permission Mapping
10-domain assessment + 1:1 permission parity verification (22→22, zero loss)
Agents 02, 03 +
Agent 02 — Gap Analysis
  • Assess 10 domains: auth_integration, session_management, account_lifecycle, platform_policies, cpm_readiness, safe_structure, permission_model, nhi_handling, integration_architecture, audit_compliance
  • Score each domain (0–5 maturity) with Privilege Cloud-specific checks
  • Map findings to 4 compliance frameworks (PCI-DSS, NIST 800-53, HIPAA, SOX)
  • Identify Cloud CPM connectivity requirements per target network segment
  • Flag Connector Server placement needs for on-prem target access
Agent 03 — Permission Verification (1:1 Parity)
  • Pull all safe members via GET /Safes/{name}/Members
  • Verify all 22 CyberArk permissions map 1:1 to Privilege Cloud — same permission model, same ACL structure
  • No permissions lost, no escalation risk, no role compression
  • Validate safe member configurations are compatible with Cloud Vault
  • Generate per-safe permission audit report confirming parity
✓

1:1 Permission Parity — Zero Loss

All 22 CyberArk safe permissions map directly to Privilege Cloud with no loss, no compression, and no escalation risk. This is a key advantage over cross-vendor migration options where granular permissions are lost.

UseAccounts • RetrieveAccounts • ListAccounts • AddAccounts • UpdateAccountContent • UpdateAccountProperties • DeleteAccounts • RenameAccounts • ManageSafe • ManageSafeMembers • ViewSafeMembers • ViewAuditLog • BackupSafe • UnlockAccounts • AccessWithoutConfirmation • CreateFolders • DeleteFolders • MoveAccountsAndFolders • SpecifyNextAccountContent • InitiateCPMAccountManagementOperations • RequestsAuthorizationLevel1 • RequestsAuthorizationLevel2
Key Advantage Over Cross-Vendor Migration

Because Privilege Cloud uses the same permission model as on-premises CyberArk, there is zero permission loss and zero escalation risk. Cross-vendor migrations (e.g., to Secret Server) map 22 safe permissions to 4 folder roles, though SS compensates with 200+ system role permissions for additional access control granularity.

P2
Infrastructure, Staging & Platform Validation
Deploy platform plugins, run 10-assertion staging validation, gate production migration
Agents 13, 10 +
Agent 13 — Platform Plugins
  • Validate Privilege Cloud platform policies match on-prem CyberArk platforms
  • Deploy custom connection components for non-standard platforms
  • Verify Connector Server placement for each target network segment
Agent 10 — Staging Validation
  • 10-assertion pipeline against a staging subset (not production):
  •   1. Create test safe → 2. Add safe member → 3. Import test account → 4. Retrieve password
  •   5. Verify heartbeat → 6. Check permissions → 7. Validate platform → 8. Test audit log
  •   9. Verify metadata → 10. Confirm rollback
  • Hard-blocks production — Agent 04 Migration Pipeline refuses to run in P4/P5 without a passing Agent 10 staging report
  • All staging artifacts are rolled back after validation (no permanent changes)
ARC-08 Gate

Agent 04 Migration Pipeline enforces ARC-08: it will not start production migration without both Agent 09 dependency map AND Agent 10 staging pass. This prevents migrating into an untested target environment.

P3
Safe & Policy Preparation + App Onboarding
Replicate safes, platform policies, and permissions in Privilege Cloud; onboard new applications
Agents 03, 14 +
Agent 03 — Safe Replication
  • Create safes in Privilege Cloud via POST /Safes — same safe names, same structure
  • Map platform policies via POST /Platforms/Targets — same platform IDs carry over
  • Apply permissions via POST /Safes/{name}/Members — same ACL model, identical permission set
  • Configure safe retention and logging policies (mirror on-prem settings)
  • Verify Connector Server placement for on-prem target network access via Cloud CPM
Agent 14 — Application Onboarding
  • 10-step onboarding pipeline for new application service accounts:
  • Validate request → Create safe → Configure permissions → Map platform
  • Create account → Assign engine → Set rotation → Verify heartbeat → Generate retrieval instructions
  • Application identity configured via ISPSS Identity Admin portal for CCP/Secrets Hub access
Same API Surface

Privilege Cloud uses the same PVWA REST API as on-premises CyberArk. Safe creation, platform mapping, and permission application use identical endpoints and payloads. No schema translation required.

  • 5-step pipeline (vs 7-step for cross-vendor): no folder creation step (safes already exist) and no separate heartbeat step (Cloud CPM handles automatically)
  • Minimal transform: same field names, same API surface — only base URL changes from on-prem PVWA to Privilege Cloud tenant

Migration Window Exposure

Accounts are frozen (CPM management disabled) during the migration pipeline. The watchdog timer limits maximum exposure to 120 minutes. Emergency unfreeze triggers automatically on any exception to minimize the window.

P4/P5
Heartbeat & Validation
Verify migrated accounts via Cloud CPM heartbeat and 10 validation checks
Agent 05 +
Agent 05 — Heartbeat & Validation
  • Count comparison — on-prem source accounts vs Privilege Cloud accounts (within threshold)
  • Heartbeat status — POST /PrivilegeCloud/API/Accounts/{id}/Verify via Cloud CPM (managed service, no manual config)
  • Permission mapping — Verify safe permissions are identical (22→22, no loss to validate)
  • Safe structure — Verify safe names and hierarchy preserved 1:1
  • Metadata integrity — Account properties, descriptions, and custom fields intact (same schema)
  • Group assignments — User/group → safe member permission mappings verified
  • Platform policies — Cloud CPM platform policies match on-prem configuration
  • Access patterns — Verify access control lists match source environment exactly
  • Audit continuity — Privilege Cloud audit trail populated for migrated accounts
  • Recording capability — PSM for Cloud / Secure Tunnel configured for session recording
Cloud CPM — Managed Service

Cloud CPM is a managed service in Privilege Cloud. No manual Distributed Engine or RPC configuration needed (unlike Secret Server). Heartbeat and password rotation are handled automatically once the Connector Server has network access to targets.

P6
Integration Repointing
Redirect CCP/AAM integrations to Privilege Cloud API — same vendor, simplified migration
Agents 15, 06 +
Agent 15 — Hybrid Fleet Orchestration
  • Orchestrate parallel-run period between CyberArk on-prem and Privilege Cloud
  • Manage traffic routing: gradual percentage shift from on-prem to cloud
  • Monitor both environments for discrepancies during parallel running
  • Cutover decision gate: threshold-based (default 99% success rate required)
  • Automatic rollback if failure threshold exceeded during parallel run
Agent 06 — Integration Repointing
  • Scan configured directories for CyberArk CCP/AAM patterns: GET /AIMWebService/api/Accounts?AppID=X&Safe=Y&Object=Z
  • Same vendor API surface — Privilege Cloud supports CCP-compatible endpoints. Many integrations require only a URL change.
  • CyberArk Central Credential Provider: on-prem CCP → Privilege Cloud CCP (Conjur Cloud or Secrets Hub for advanced use cases)
  • PowerShell: psPAS module works for both on-prem and Privilege Cloud — change base URL and authentication method only
  • .NET: CyberArk SDK works for both — update connection string to Privilege Cloud tenant URL
  • REST API: Same endpoint paths — /PasswordVault/API/ prefix remains the same, only base URL changes
  • Applications using certificate-based auth may need to switch to OAuth2/ISPSS identity tokens
  • Maintain change registry for tracking and rollback during parallel running period
Authentication Method Change

While the REST API surface is the same, authentication may change from CyberArk Vault auth (LDAP/RADIUS/CyberArk) to CyberArk Identity (ISPSS OAuth2). Applications using CCP certificate-based authentication need to be updated to use the new identity provider.

P7
Decommission & Close-Out
Migrate audit logs and PSM recordings, compliance sign-off, decommission on-prem infrastructure
Agent 07 +
Agent 07 — Compliance & Decommission
  • Audit logs CAN be migrated — CyberArk provides tools to transfer audit history to Privilege Cloud (advantage over cross-vendor migration)
  • PSM recordings CAN be transferred — session recordings can be migrated to Privilege Cloud Secure Tunnel / PSM for Cloud storage
  • Verify all 4 compliance frameworks have passing evidence: PCI-DSS v4.0, NIST 800-53 Rev5, HIPAA Security Rule, SOX IT Controls
  • Generate final compliance report — no permission loss to document, no audit discontinuity
  • After parallel running period, execute decommission sequence:
  •   1. PSM/PSMP servers — decommission first (sessions now handled by Privilege Cloud)
  •   2. CPM servers — decommission second (Cloud CPM now manages rotation)
  •   3. PVWA servers — decommission third (Privilege Cloud portal replaces)
  •   4. Vault server — decommission last (only after all data confirmed migrated)
  • Final sign-off gate (human approval required at each decommission step)
Audit & Recording Continuity

Unlike cross-vendor migrations, CyberArk-to-Privilege Cloud migration preserves audit log continuity and PSM session recordings. No data loss and no need to maintain a read-only CyberArk instance for audit compliance.

Migration Risks — Privilege Cloud Specific
MEDIUM

Migration Window Exposure

Accounts are frozen (CPM management disabled) during migration pipeline execution. If the migration stalls, accounts remain unmanaged until manually unfrozen.
Mitigation
Watchdog timer (120 min default) automatically unfreezes all frozen accounts on timeout. Emergency unfreeze triggers on any exception before re-raising.
MEDIUM

Cloud CPM Connectivity

Cloud CPM needs a network path to on-prem targets for heartbeat and password rotation. Firewall rules and Connector Server placement are critical.
Mitigation
Deploy Connector Servers in each target network segment. Validate connectivity per segment before wave migration begins. Cloud CPM health checks confirm reachability.
MEDIUM

NHI Dependency Chain

Applications using CCP/AAM need repointing to Privilege Cloud endpoints. NHI accounts with active integrations must be migrated carefully to avoid service disruption.
Mitigation
NHIs with CCP/AAM integrations are migrated in Wave 5 (last). Parallel running period allows gradual integration cutover. Same-vendor API surface reduces repointing complexity.
