Lifecycle Phases
P1
Discovery & Dependency Mapping
Scan CyberArk on-prem to inventory accounts, safes, platforms, and NHIs
Agents 11, 01, 09, 12 +
Agent 11 — Source Adapter
  • Normalize accounts from any source vendor (CyberArk, BeyondTrust, HashiCorp, AWS, Azure, GCP) into canonical format
  • Provides unified NormalizedAccount schema regardless of source PAM vendor
Agent 01 — Discovery
  • Connect to CyberArk PVWA via GET /Accounts (paginated, count-based)
  • Enumerate all safes via GET /Safes — filter system safes (System, VaultInternal, Notification Engine, etc.)
  • Enumerate platforms via GET /Platforms/Targets
  • NHI Detection (multi-signal): platform-based (UnixSSHKeys, WinServiceAccount, AzureServicePrincipal), name patterns (^svc[_-], ^app[_-], ^api[_-]), safe name patterns (appcred, automation, cicd)
  • Scan for CCP/AAM integrations via GET /Applications + GET /Applications/{id}/Authentications
  • SS Template Gap Detection: Cross-reference discovered CyberArk platforms against known Secret Server templates to identify custom platforms that need manual template creation
  • Output discovery manifest with account counts, safe inventory, NHI list, integration map, and template gaps
Agent 09 — Dependency Mapper
  • Build dependency graph: application → account → safe → platform mappings
  • Scan IIS logs and config files for CCP/AAM connection strings
  • Identify NHI dependency chains that must migrate together
  • Produce migration ordering constraints for Agent 04 Migration Pipeline wave classification
Agent 12 — NHI Handler
  • Weighted multi-signal NHI classification: platform (0.35), name pattern (0.20), container context (0.10), dependency graph (0.25), audit frequency (0.10)
  • Risk scoring: confidence threshold (default 0.6) determines NHI vs human classification
  • Generate rotation policies based on NHI type (service account, API key, certificate)
P1
Gap Analysis & Permission Translation
10-domain assessment + 22→4 folder role translation + 200+ system role permissions with escalation detection
Agents 02, 03 +
Agent 02 — Gap Analysis
  • Assess 10 domains: auth_integration, session_management, account_lifecycle, secret_templates, rpc_readiness, folder_hierarchy, permission_model, nhi_handling, integration_architecture, audit_compliance
  • Score each domain (0–5 maturity) with SS-specific checks
  • Map findings to 4 compliance frameworks (PCI-DSS, NIST 800-53, HIPAA, SOX)
  • Identify SS template coverage gaps (CyberArk platforms without matching SS templates)
  • Flag RPC readiness — Distributed Engine availability per target network segment
Agent 03 — Permission Translation (Folder + Role Mapping)
  • Pull all safe members via GET /Safes/{name}/Members
  • Map each member's 22 CyberArk safe permissions to 4 SS folder roles (Owner, Edit, Add Secret, View) + assign granular system role permissions from 200+ available
  • Track 9 permissions with NO SS equivalent — always lost
  • Detect escalation risks: members who get MORE access than they had (e.g., View→Edit when only UnlockAccounts was granted)
  • Generate per-member loss report with assigned role, lost permissions, and risk flags

Escalation Risk

A member with only UseAccounts + RetrieveAccounts + UnlockAccounts gets the Edit role (full create/modify/delete) because UnlockAccounts triggers Edit. They now have more access than before.

P2
Infrastructure, Staging & Template Validation
Validate templates, deploy RPC components, run 10-assertion staging validation, gate production
Agents 13, 10 +
Agent 13 — Platform Plugins
  • Validate Secret Server template coverage against discovered CyberArk platforms
  • Deploy RPC (Remote Password Changing) connection components per template
  • Verify Distributed Engine placement for each target network segment
Agent 10 — Staging Validation
  • 10-assertion pipeline against a staging subset (not production):
  •   1. Create test folder → 2. Set folder permission → 3. Create test secret → 4. Retrieve password
  •   5. Heartbeat (RPC) → 6. Check permissions → 7. Validate template → 8. Test audit log
  •   9. Verify metadata → 10. Confirm rollback
  • Hard-blocks production — Agent 04 Migration Pipeline refuses to run in P4/P5 without a passing Agent 10 staging report
  • All staging artifacts are rolled back after validation (no permanent changes)
ARC-08 Gate

Agent 04 Migration Pipeline enforces ARC-08: it will not start production migration without both Agent 09 dependency map AND Agent 10 staging pass. This prevents migrating into an untested target environment.

P3
Folder & Template Preparation + App Onboarding
Create folder hierarchy in Secret Server, map platforms to templates, apply permissions; onboard new applications
Agents 03, 14 +
Agent 03 — Permission Mapping (Apply Phase)
  • Create parent /Imported/ folder via POST /api/v1/folders
  • Create per-safe child folders: /Imported/{SafeName}/ (hierarchical, not flat)
  • Configure folder permission inheritance from parent where appropriate
  • Apply translated permissions via POST /api/v1/folder-permissions (folderAccessRoleName + secretAccessRoleName)
  • Map CyberArk platforms to SS templates (manual template creation for custom platforms identified in P1)
Platform → Template Mapping

WinServerLocal → Windows Account, WinDomain → Active Directory Account, UnixSSH → Unix Account (SSH), UnixSSHKeys → Unix Account (SSH Key Rotation), Oracle/MSSql/MySQL → corresponding DB templates. Custom CyberArk platforms require manual SS template creation before the migration pipeline can proceed.

Agent 14 — Application Onboarding
  • 10-step onboarding pipeline for new application service accounts:
  • Validate request → Create folder → Set permissions → Resolve template
  • Create secret → Assign engine → Set rotation → Heartbeat (RPC verify) → Generate retrieval instructions
  • Auto-change must be configured via SS Admin UI (no dedicated API method)

P4/P5
Heartbeat & Validation
Verify migrated secrets are functional via SS heartbeat API and 10 validation checks
Agent 05 +
Agent 05 — Heartbeat & Validation
  • Count comparison — CyberArk source accounts vs Secret Server secrets (within threshold)
  • Heartbeat status — POST /api/v1/secrets/{id}/heartbeat — check LastHeartbeatStatus via RPC
  • Permission mapping — Verify folder permissions match translated roles; flag 22→4 escalation risks
  • Folder structure — Verify /Imported/{SafeName} hierarchy preserved
  • Metadata integrity — Descriptions and custom fields intact (properties preserved in notes field)
  • Group assignments — User/group → folder permission role mappings verified
  • Password policies — RPC (Remote Password Changing) configured per template
  • Access patterns — Flags >30% Edit/Owner roles as potential over-provisioning
  • Audit continuity — SS audit trail populated for migrated secrets
  • Recording preservation — PSM recordings archived (NOT migrated to SS)
Audit Log Discontinuity

CyberArk audit history does NOT transfer to Secret Server. There will be a gap in the audit trail. Maintain CyberArk in read-only mode for the audit retention period.

RPC Configuration

RPC (password rotation) requires manual Distributed Engine configuration per template. This is verified here but must be set up during Phase P2 (Infrastructure Preparation).

P6
Integration Repointing
Re-architect CCP/AAM integrations to use Secret Server REST API + OAuth2
Agents 15, 06 +
Agent 15 — Hybrid Fleet Orchestration
  • Orchestrate parallel-run period between CyberArk on-prem and Secret Server
  • Manage traffic routing: gradual percentage shift from CyberArk to SS
  • Monitor both environments for discrepancies during parallel running
  • Cutover decision gate: threshold-based (default 99% success rate required)
  • Automatic rollback if failure threshold exceeded during parallel run
Agent 06 — Integration Repointing
  • Scan configured directories for CyberArk CCP/AAM patterns: GET /AIMWebService/api/Accounts?AppID=X&Safe=Y&Object=Z
  • This is NOT a URL swap — the authentication model (OAuth2 vs AppID), endpoint structure, and data format are completely different
  • Generate language-specific replacement code templates: Python, PowerShell, C#, Java
  • New pattern: POST /oauth2/token (client_credentials) → GET /api/v1/secrets/{id}/fields/password
  • PowerShell: psPAS / Get-PASAccount → Thycotic.SecretServer / Get-TssSecret
  • .NET: CyberArk.AIM.NetPasswordSDK → HttpClient + OAuth2 + /api/v1/secrets
  • Maintain change registry for tracking and rollback
Full Re-Architecture Required

Every application using CyberArk CCP/AAM must be rewritten. There is no backward compatibility layer. Plan for developer time proportional to integration count.

P7
Decommission & Close-Out
Archive CyberArk audit logs and PSM recordings, compliance sign-off, decommission
Agent 07 +
Agent 07 — Compliance & Audit
  • Archive CyberArk audit logs — they do NOT transfer to Secret Server. Export via GET /Activities and store for retention period.
  • Archive PSM session recordings — cannot be migrated to SS Protocol Handler. Maintain read-only CyberArk access for recording review.
  • Verify all 4 compliance frameworks have passing evidence: PCI-DSS v4.0, NIST 800-53 Rev5, HIPAA Security Rule, SOX IT Controls
  • Generate final compliance report with SS-specific risk documentation (permission loss, audit discontinuity, recording loss)
  • After parallel running period, schedule CyberArk for decommission
  • Final sign-off gate (human approval required)
CyberArk Read-Only Retention

Keep CyberArk running in read-only mode for the audit retention period (typically 1–7 years depending on compliance framework). Do not delete until all audit obligations are met.

Migration Risks — Secret Server Specific
HIGH

Audit Log Discontinuity

CyberArk audit history does NOT transfer to Secret Server. There will be a gap in the audit trail during and after migration.
Mitigation
Maintain CyberArk in read-only mode for audit retention period. Document the gap for auditors. Export audit logs via API before decommission.
HIGH

Permission Model Simplification

22 CyberArk safe permissions map to 4 Secret Server folder roles. At the folder level, some safe-specific permissions have no direct equivalent. However, SS provides 200+ system role permissions for additional granularity. Escalation risk exists where folder role mapping grants broader access than the original safe permissions.
Mitigation
Review Agent 03 permission loss report before P3. Document all escalations. Consider SS Workflow templates for dual-control equivalence.
MEDIUM

PSM Session Recording Loss

PSM session recordings cannot be migrated to Secret Server's Protocol Handler. Historical recordings will be inaccessible from SS.
Mitigation
Archive all PSM recordings before CyberArk decommission. Maintain read-only CyberArk access for recording review during retention period.
